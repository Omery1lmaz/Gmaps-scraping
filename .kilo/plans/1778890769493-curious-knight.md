# Plan: Update Dashboard Main Page (Overview) to Remove Mock Data

## Goal
Replace hardcoded mock data in the main Overview page (`dashboard/src/pages/Overview.tsx`) with real, dynamic data fetched from the backend API. The current implementation uses static arrays for chart data, stats, and recent activity. Other pages (LeadsPage, etc.) already use real API via react-query, so align Overview with that pattern.

## Current Issues Identified
- `dashboard/src/pages/Overview.tsx:23-31`: Hardcoded `data` array for AreaChart (leads/contacted by day).
- `dashboard/src/pages/Overview.tsx:33-38`: Hardcoded `stats` with fake numbers (Total Leads: 1,284, etc.).
- `dashboard/src/pages/Overview.tsx:96-106`: Hardcoded recent activity loop with fake names like "Kebapçı İsmet".
- No use of `useQuery` or API calls (unlike LeadsPage.tsx:53-56).
- MOCK_USER_ID constants exist in other pages (WhatsAppPage, etc.) but not directly in main Overview.
- Backend (`server/index.ts`) provides `/api/leads` but no dedicated `/api/stats` or analytics endpoint for aggregates.

## Proposed Changes
1. **Update `dashboard/src/pages/Overview.tsx`**:
   - Import `useQuery` from '@tanstack/react-query' and `getLeads` from '../lib/api'.
   - Fetch leads with `useQuery({ queryKey: ['leads'], queryFn: getLeads })`.
   - Compute real stats dynamically:
     - Total Leads: `leads.length`.
     - Active Pipeline: count where status !== 'CLOSED' && !== 'REJECTED'.
     - Conversion Rate: (closed / total * 100).toFixed(1) + '%'.
     - Closed Deals: count where status === 'CLOSED'.
   - For chart data: Group leads by day-of-week (use createdAt or a fallback date field from Lead model), aggregate counts for leads + contacted statuses. Use date-fns or native JS for grouping.
   - Replace static recent activity with top 5 recent leads (sorted by createdAt desc), showing name + time ago (use date-fns or dayjs for formatting).
   - Add loading states (Skeleton or isLoading check) for stats/cards/chart.
   - Keep existing UI/layout/charts structure intact.

2. **Enhance API if needed** (minimal):
   - Optionally add `getLeadsStats` helper in `dashboard/src/lib/api.ts` (but prefer client-side computation from getLeads for simplicity, as data volume is low).
   - No backend changes required initially.

3. **Dependencies**:
   - Ensure `date-fns` or similar is available (check package.json; use native if possible to avoid new deps).
   - React-query already used in LeadsPage.

4. **Testing/Validation**:
   - After update, verify Overview renders with live data from running server (localhost:3001).
   - Ensure no regressions in other views.

## Implementation Steps (for execution phase)
- Read full Overview.tsx + related files (already partially done).
- Implement fetch + compute logic in Overview.
- Use `suggest` after for local review if changes are non-trivial.

## Tradeoffs Considered
- Client-side aggregation vs new backend endpoint: Prefer client-side to keep changes minimal (no server edits).
- Date grouping: Assume leads have `createdAt`; fallback to current if missing (per Lead.ts model).
- No questions needed; plan is straightforward based on existing patterns in LeadsPage.tsx.

This plan ensures the dashboard main page becomes fully dynamic and consistent with the rest of the app.
